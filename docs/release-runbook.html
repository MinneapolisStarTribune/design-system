Release Runbook

Design System — MinneapolisStarTribune · GitHub Packages · npm scope: @minneapolisstartribune · Maintained by: Platform / Shared UI

00Overview

This document is the single source of truth for cutting a release of @minneapolisstartribune/design-system. It covers both feature releases (new components, API changes) and hotfix releases (urgent patches to a shipped version), including the decision of when to actually publish a package to GitHub Packages and when to absorb the work into the next scheduled release.

All release automation runs through GitHub Actions. Manual steps require a person with repository write access. Steps marked auto fire without intervention once the preceding manual step is complete.

ℹ Key rule: Only branches named release/* may merge into main. Feature branches, hotfix branches, and any other work must target a release branch first. This is enforced by CI and will block the PR if violated.

01Release Types

We use two categories of release. The version numbers follow semver, but internally we call them "feature" and "hotfix" to be explicit about intent.

Feature Release
e.g. v1.11.0 → v1.12.0
Scheduled releases containing new components, design token updates, or any breaking/non-breaking API changes.
  • Branch cut from main
  • Multiple feature PRs may land before merge
  • Always published to GitHub Packages
  • Patch digit is 0 (e.g., v1.12.0)
Hotfix / Patch Release
e.g. v1.11.0 → v1.11.1
Targeted fixes to a version already in production. Does not always result in a published package — the work may be absorbed into the next feature release instead.
  • Branch cut from main (at the tag commit)
  • Typically one or two focused PRs
  • Optional publish — see When to Publish
  • Patch digit is > 0 (e.g., v1.11.1)
⚠ Terminology note: In conversation, the team sometimes calls feature releases "major" and hotfixes "minor." This does not map to semver major/minor — the package is still at major version 1. Use "feature release" and "hotfix" when communicating to avoid ambiguity.

02Feature Release Flow

Follow these steps in order when cutting a scheduled feature release (e.g., v1.12.0).

1
Create the release branch Manual

Go to Actions → "Create release branch" → Run workflow.

  • release_version: enter 1.12.0 or v1.12.0 (both accepted)
  • base_branch: leave as main (default)

The workflow creates release/1.12.0 from the tip of main and pushes it. If the branch already exists, it exits cleanly without error.

2
Develop against the release branch Manual

Cut feature branches from release/1.12.0 — not from main. Open PRs that target release/1.12.0.

  • 2 approvals required before merge
  • Squash or merge commit — no rebase
CI
PR checks run automatically Auto
  • lint.yml — ESLint, Prettier, TypeScript typecheck
  • component-tests.yml — unit tests + a11y tests with Codecov upload
  • chromatic.yml — visual regression (runs only if .stories.ts/tsx files or Storybook config changed)
3
Open PR: release branch → main Manual

When all features are merged into release/1.12.0 and the release is ready to ship, open a PR from release/1.12.0 → main.

The merging-to-main-restriction check will pass automatically because the source branch starts with release/. All other PR checks also run on this PR. 2 approvals required.

4
Merge the PR into main Manual

Approve and merge. Squash or merge commit — match what the team used for feature PRs on this release.

↻
Back-merge PRs created for higher release branches Auto

back-merge-main-to-release.yml fires immediately after merge. It finds any release branches with a version number higher than the one just merged (e.g., if release/1.12.1 exists), and opens a PR from main → that branch. Review and merge these promptly to avoid divergence.

5
Create a GitHub Release and tag Manual — last manual step

Go to Releases → "Draft a new release".

  • Tag: v1.12.0 — must match vX.Y.Z exactly
  • Target: main — always tag against main, never a release branch
  • Write release notes and click Publish release

Clicking Publish release is the trigger. Everything after this — the npm publish, the version bump, the Slack post — happens automatically. No further action is needed.

⚠ Tag caution: "Restrict creations" is currently unchecked in branch protection — any team member can push a tag. An accidental tag push triggers the publish pipeline immediately. Coordinate before creating tags. Always double-check that main is in the correct state before publishing the release.
↻
Publish pipeline runs automatically — no action required Auto

publish.yml fires immediately on the release: published event:

  • Checks out main (not the tag ref — the code published always comes from main)
  • Validates tag format — fails early if not vX.Y.Z
  • Runs release:verify gates (web, native, a11y)
  • Regenerates and commits the component matrix to main if changed
  • Updates packages/design-system/package.json version → commits "chore: bump version to 1.12.0" → pushes to main
  • Builds and publishes to GitHub Packages as @minneapolisstartribune/design-system@1.12.0

Simultaneously, release-changelog.yml generates a changelog from merged PR titles and posts it to the Slack release channels.

ℹ Manual escape hatch: If the pipeline fails and needs to be re-run without creating a new release, go to Actions → "Publish Design System" → Run workflow and enter the version manually. This is the workflow_dispatch path — only use it for reruns, not for normal releases.

03Hotfix / Patch Release Flow

Use this flow when a critical bug in a shipped version needs to be addressed. The process is nearly identical to a feature release, with one additional decision point: whether the patch should ship as its own version or be absorbed into the next feature release.

1
Create the hotfix release branch Manual

Go to Actions → "Create release branch".

  • release_version: 1.11.1 (the patch version)
  • base_branch: main — since v1.11.0 was already merged and tagged on main, the fix should start from there

Creates release/1.11.1 from the tip of main.

2
Develop the fix Manual

Cut fix branches from release/1.11.1. PRs target release/1.11.1. Same CI gates and 2-approval requirement apply.

?
Decide: standalone patch or absorption? Decision

This is the key judgment call. See When to Publish for the criteria. Two paths:

  • Publish as v1.11.1 — the fix is urgent and consumers need it before the next feature release. Continue to steps 3–4.
  • Absorb into next feature release — the fix can wait, or is already included in an in-progress feature release branch. Merge release/1.11.1 directly into the next feature release branch (e.g., release/1.12.0). v1.11.1 is never tagged. Skip to that release's flow.
3
Merge hotfix branch into main and publish Manual

Only if publishing as a standalone patch.

Open PR from release/1.11.1 → main, get 2 approvals, merge. Then go to Releases → "Draft a new release", create tag v1.11.1 targeting main, and click Publish release. That click is the only action required — the npm publish, version bump, and Slack changelog all run automatically from there.

↻
Back-merge and publish (same as feature release) Auto

After merge: back-merge workflow fires, creating PRs from main into any higher release branches (e.g., release/1.12.0 if it exists). After tag + release publish: publish.yml and release-changelog.yml fire identically to a feature release.

04Absorption Pattern

The most recent example: hotfix work was done in release/1.10.1, but v1.10.1 was never published. The work was absorbed into v1.11.0, which is the version consumers received.

ℹ When a hotfix is absorbed, the hotfix release branch (release/1.10.1) still existed and was used for development — it was just never merged to main and never tagged. The fix landed in the next feature release branch instead.
Absorption steps
  1. Fix is developed in release/1.10.1 normally
  2. Instead of opening a PR to main, open a PR from release/1.10.1 → release/1.11.0
  3. The merging-to-main-restriction check does not run (target is not main)
  4. Normal CI and 2-approval requirement still apply
  5. After merge, the fix is part of release/1.11.0 and ships with v1.11.0
  6. v1.10.1 is never created — it does not exist in GitHub Packages
⚠ Consumers who pinned to v1.10.0 will not automatically receive the hotfix — they need to upgrade to v1.11.0. If the fix is truly urgent and consumers cannot wait for the feature release, publish the standalone patch instead of absorbing it.

05When to Publish

Publishing to GitHub Packages is a one-way action — you cannot unpublish a version. Use this table to make the call before creating the GitHub Release.

ScenarioPublish?Tag created?Notes
Feature release (e.g., v1.12.0) AlwaysYes No exceptions. Feature releases must ship to consumers.
Hotfix — urgent, needed now YesYes Consumers cannot wait for the next feature release. Ship the patch.
Hotfix — absorbed into next feature release NoNo Fix lands in the feature release. Hotfix branch discarded after merge.
Hotfix — next feature release ships soon Your callMaybe If the feature release is days away, absorbing is usually right. If it's weeks, publish the patch.

06Tag Rules

Required format
vX.Y.Z
Target branch
main
Quick links
All tags → All releases → New release →

Tags are validated by the validate-tag-format composite action, which rejects anything that doesn't match ^v[0-9]+\.[0-9]+\.[0-9]+$. A publish triggered by an invalid tag will fail early at validation.

The publish workflow always checks out main — not the tagged commit. This means the package content published to GitHub Packages always reflects the state of main at the moment the workflow runs, regardless of which commit the tag points to.

⚠ "Restrict creations" is unchecked. GitHub branch protection currently does not restrict who can create tags. Any team member with write access can push a vX.Y.Z tag and trigger the full publish pipeline unintentionally. Coordinate with the team before creating tags, and verify that main is in the intended state immediately before publishing a release.
Before publishing a release — pre-flight checklist
  • The release branch PR has been merged into main
  • All CI checks on main are green
  • The intended version number does not already exist — check the tags list
  • Any back-merge PRs from the previous step are open and acknowledged
  • The tag is being created against the main branch, not a release branch

07CI/CD Workflow Reference

All workflows live in .github/workflows/. The table below summarizes what fires, when, and what it does.

Workflow fileTriggerWhat it does
merging-to-main-restriction.yml Any PR Fails the PR if the target is main and the source branch does not start with release/.
lint.yml PR to main or release/** Runs ESLint, Prettier format check, and TypeScript typecheck.
component-tests.yml PR to main or release/** Runs unit tests and a11y tests; uploads coverage and results to Codecov.
chromatic.yml PR to main or release/** Visual regression via Chromatic. Only runs if .stories.ts/tsx files or .storybook/ config changed.
create-release-branch.yml Manual dispatch Creates release/X.Y.Z from a specified base branch. Validates semver format. Exits cleanly if branch already exists.
publish.yml GitHub Release published or manual dispatch Checks out main; validates tag; runs release:verify; bumps version in package.json; commits and pushes to main; builds; publishes to GitHub Packages.
release-changelog.yml GitHub Release published or manual dispatch Walks merged PRs since the previous tag; categorizes by title keywords; posts formatted changelog to Slack release channels.
back-merge-main-to-release.yml A release/* branch is merged into main Finds all active release branches with a higher version number and opens PRs from main into each. Notes merge conflicts in the PR body when detected.
sync-versions-from-vercel.yml Schedule / manual Syncs Storybook version metadata from Vercel production deployments.

09Future State — v2.0.0

For the next major version, the team is evaluating conventional commits and semantic release tooling to automate the parts of this workflow that are currently manual. Nothing below is decided — it's a direction to investigate.

Conventional Commits
Standardize commit messages with prefixes: feat:, fix:, chore:, BREAKING CHANGE:. This is the prerequisite for all automation below. Tools: commitlint + a lefthook commit-msg hook (lefthook is already in the repo).
Automated versioning
Tools like semantic-release, changesets, or release-please can determine the next version number from commit history and create the GitHub Release automatically. This eliminates the manual tag creation step and its associated risks.
Automated changelog
With conventional commits, release-changelog.yml can derive accurate categorized changelogs from commit messages rather than PR title heuristics — removing false positives from keyword matching.
Consideration: opinionated vs. flexible
semantic-release is more opinionated and fully automated. changesets is more flexible and keeps humans in the loop for version decisions. Given the current team size and the "optional publish" pattern for hotfixes, changesets may be the better fit — but this is worth a dedicated decision meeting before v2.0.0.
